我给自己做了一个“返回来源标签页”的小扩展，不是因为浏览器不能打开新标签页，而是因为它太会打开了，却不太会帮我保留“我是从哪里跳过来的”这件事。

平时浏览网页时，我经常会从一个页面点开链接，进入一个新标签页。内容当然是顺利打开了，但每次进入新标签页之后，我都会有一种很轻微却很持续的不适感：页面到了，路径却断了。浏览器提供的“返回”按钮，只能让我回到同一个标签页里的上一页，却不能让我回到“打开这个新标签页之前，我所在的那个标签页”。这两种“返回”看起来很像，实际上不是一回事。前者处理的是页面历史，后者处理的是来源关系；前者关心我浏览过什么，后者关心我是从哪里来的。

我真正想解决的，就是这个被忽略的断层。

很多时候，标签页系统已经默认我们会频繁跳转、频繁打开新页面，但它并没有认真处理这些跳转之间的上下文关系。一个新标签页被打开后，在浏览器内部，它只是一个新的容器；但在用户的认知里，它不是凭空出现的，它有明确的来路。它是我从某个页面中点开的，是我在某个上下文里决定进一步展开查看的内容。可一旦这个页面被放进新标签页，这条“来路”通常就消失了。于是用户只能自己记住：我是从哪里点开的、之前看到了哪里、等会儿还要不要回去。这个负担很小，但它出现得足够频繁，久而久之就会变成一种浏览时的摩擦感。

我做这个小扩展的出发点，其实很简单：我想让“打开新标签页”这件事，不再等于“丢失原来的位置”。我希望当一个页面是从另一个标签页打开的时候，它们之间的关系能被保留下来；这样当我看完新内容后，不需要重新找，不需要切换一堆标签，也不需要在脑子里记住刚才的入口，只要点一下按钮，就能回到打开它之前所在的那个标签页。

所以，这个扩展并不是在重复浏览器已经有的“后退”功能。它关心的不是同一个页面里的前进后退，而是不同标签页之间的来源关系。换句话说，我想要的不是 page back，而是 context back。不是“回到上一个网页”，而是“回到打开这个网页之前的那个位置”。这听起来只是一个很小的交互差异，但对实际体验来说，它影响的是方向感、连续性和认知负担。

我也不希望这个工具变成一个很重、很复杂的系统。它不需要侧边栏，不需要额外的面板，不需要改造整个标签页工作流。它只需要安静地待在顶部工具栏里，像浏览器原本就应该提供的一个小功能：当当前标签页有明确来源时，它就亮起来，告诉我“你可以回去”；当没有来源时，它就变灰，安静地不打扰。这种状态变化本身也是我很看重的一点，因为我不想让工具替用户做太多判断，而是想用一种尽量克制的方式，让信息和操作保持一致。

对我来说，这个扩展的价值，不在于它有多强大，而在于它是否足够顺手。它解决的不是一个宏大的效率问题，而是一种真实、细小、反复出现的使用不适：内容已经被打开，但上下文没有被保留；跳转已经完成，但路径感消失了。我想做的，就是把这条路径补回来。

如果要用一句话概括这个扩展的创作理念，那就是：**浏览器不应该只记住我看到了什么，也应该记住我是从哪里来到这里的。​**
